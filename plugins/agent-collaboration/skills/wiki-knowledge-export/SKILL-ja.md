# wiki 知見の書き出し手順書

発生元ブランチが消えても有用な知見=**永続知見**を GitHub wiki(別 git リポジトリ)に書き出し、`Home.md` 索引を同期する。

これは**永続層**である。対になるのは `conversation-context-export`(**揮発層**: ブランチ/PR 単位の文脈を `.dev/contexts/` + PR コメントへ)。以下の判定で発見を正しい層へ振り分ける。

## 1. 判定: 永続か揮発か

以下の**3条件すべて**を満たすときだけ、その発見を**永続**(wiki 行き)とみなす:

1. 特定の変更ではなく、**プロジェクト/システム/ツール自体**についての知見である
2. 発生元ブランチが merge または破棄された**後も**有用である
3. **無関係な別ブランチの将来セッション**が恩恵を受ける、または放置すると再発見してしまう種類である

鋭い判別テスト:

- **ブランチ削除テスト**: 「このブランチが半年後に消えても、これは要るか?」要る→永続。
- **タイトルテスト**: 自然なタイトルが**話題/概念**(永続)か、**ブランチ/PR/変更**(揮発)か。
- **種別テスト**: 実測値、ツール評価、アーキテクチャ判断(ADR)、規約・方針、システム横断の落とし穴→永続。特定変更の根拠、*この* PR の却下代替案、*この*ブランチの残作業、一度きりのデバッグメモ→揮発。

振り分け:

- **揮発** → この skill は使わない。`conversation-context-export` を使う。
- **混在**(特定変更の調査だが永続的な事実も得られた)→ 分割する: 永続的な事実は wiki へ、変更の根拠は `conversation-context-export` へ。
- **不明** → 発見と判定をユーザーに提示し、どちらの層か尋ねる。

## 2. wiki の所在を特定する

Bashツールで以下を実行する:

```
gh repo view --json url -q .url
```

出力を `{repo-url}` とする(例: `https://github.com/owner/repo`)。すると:

- wiki の git remote: `{repo-url}.wiki.git`
- ページの web URL: `{repo-url}/wiki/{PageName}`(`.md` を除いたファイル名)

wiki に到達できるか確認する:

```
git ls-remote {repo-url}.wiki.git
```

- ref が列挙されれば続行。
- エラーまたは何も出ない場合、wiki は未初期化(Wikis 無効、または最初のページが未作成)。ユーザーに報告する: 「リポジトリの Wikis を有効化し、最初のページを web UI で1度だけ作成してから再実行してください」。停止する——未初期化の wiki は push では作れない。

## 3. wiki を clone する

スクラッチディレクトリ(メインリポジトリの作業ツリーの外)へ clone する:

```
git clone {repo-url}.wiki.git {scratch}/repo.wiki
```

セッションのスクラッチディレクトリ配下のパスを使う。ユーザーの git が使う remote プロトコルに合わせる(`gh auth status` で ssh / https を確認)。`gh repo view` は https を返すので、ユーザーの git が ssh なら `git@github.com:owner/repo.wiki.git` を使う。

## 4. ページを選び、下書きする

### ページ名

既存ページの作法(例: `SKILL-token-ja-en.md`、`waza-check.md`)に合わせ、**話題で名付けた**ファイルにする。Title Case か kebab、語は `-` でつなぐ。ページ名は変更ではなく概念として読めること。

### 新規 vs 更新

clone 内に `{PageName}.md` が既に存在するか確認する。

- **新規ページ**: 作成する。
- **既存ページ**: まず読み込み、以下のルールで更新する(wiki は本質的にセッション横断、場合により著者横断):
  - 既存の項目は原則残し、新情報を追記する。
  - **実際に追試して反証したものだけ**を修正・削除する。推論だけでは削除しない。
  - ページは常に現時点で正しい状態を表す。履歴は git に任せる(ページ内に取り消し線・変更履歴を残さない)。

### ページ本文

この skill のディレクトリの [TEMPLATE.md](TEMPLATE.md) を読み、それに従う。要点:

- ページを**自己完結**させる: 発生元ブランチは消えるので、単独で成立する文脈を含める。
- 実測・評価は**日付と出典コマンド/コミット**を記録し、後から陳腐化が分かるようにする。
- `conversation-context-export` と同じ「重点/薄くてよい」の規律を適用する: 永続的な事実・判断とその根拠であって、苦労話やコードを読めば自明なことは書かない。

## 5. Home.md 索引を更新する

`Home.md` を開き、ページ一覧の見出し(例: `## ページ一覧`)の下に、既存の形式で1行のポインタを追加する(未記載の場合):

```
- [[{PageName}]] — {そのページが何を記録し、なぜ重要かの一言}
```

既存ページの更新では、スコープが変わったときだけ一言を更新する。

## 6. 確認してから commit & push

wiki への書き込みは外部公開である。**push の前に**、下書きしたページ(と `Home.md` の変更)をユーザーに提示して確認する——ユーザーが既に「確認なしで push」と言っていない限り AskUserQuestion ツールを使う。

確認後、clone 内で:

```
git add {PageName}.md Home.md
git commit -m "{簡潔なメッセージ}"
git push
```

## 7. 報告

ページの web URL(`{repo-url}/wiki/{PageName}`)をユーザーに報告する。発見を分割した場合は、`conversation-context-export` へ回した内容も報告する。

## 関連スキル

- **conversation-context-export**: 揮発層——ブランチ/PR 単位の文脈を `.dev/contexts/` + PR コメントへ。揮発する発見はこちらへ回す。
- **conversation-context-import**: 保存した揮発文脈を読み込む。
