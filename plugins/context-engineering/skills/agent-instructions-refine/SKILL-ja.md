# Refine CLAUDE.md(エージェント指示ファイルのスリム化)

CLAUDE.md / AGENTS.md は毎ターン読まれる。
肥大したファイルはトークンを食うだけでなく、本当に効かせたい規則を埋もれさせ、結果として無視される。
小さく密度高く保つ:広く適用されるものと、agent が自力で導けないものだけを残し、それ以外は外へ出す。

常時ロードされるエージェント指示ファイル(CLAUDE.md・AGENTS.md とその同等物)に適用する。

## 判定基準

各行について問う:**「これを削除すると agent が何かを間違えるか?」**
そうでなければ削る。
以下の keep/cut 判断はすべてこの一問で決まる。

## 残す vs 削る

| 残す(導けない・広く適用される) | 削る(導ける・一般的・揮発的) |
| --- | --- |
| agent が当てられない bash コマンド — 説明でなく**正確な実行形**(`pytest -v --no-header`。「テストを走らせる」ではなく) | コードを読めば agent が学べること |
| 既定と異なるコードスタイルを**具体的な規則で**(「検証は Zod、生の型は使わない」「API ルートは `/lib/auth` 経由」) | agent が既に知る標準的な言語慣習 |
| テスト指示、推奨テストランナー、フィクスチャの置き場所 | 詳細な API ドキュメント(リンクで示す) |
| リポジトリの作法:ブランチ命名、PR / コミット規約 | 頻繁に変わる情報 |
| プロジェクト固有のアーキテクチャ決定 | 長い説明・チュートリアル・ファイル単位の解説 |
| 標準から外れたディレクトリ構造 / 規約 — 外れている点だけ。ファイル単位の羅列ではなく | 自明な助言(「きれいなコードを書く」) |
| **所有権境界** — モノレポで各サービス / モジュールが何を所有し、何を**所有しない**か | **シークレット・認証情報**(API キー、パスワード、トークン) — private repo でも絶対に書かない |
| 開発環境の癖(必須の環境変数、`.env.example` の場所)、非自明な落とし穴、見落としやすい追従編集 | 実際には従っていない理想の規約 — agent が厳格適用して摩擦を生む。**やっていること**を書く |
|  | 維持する気のない陳腐化した記述 — agent は書かれた規則を自信を持って実行するので、古い指示は積極的に誤らせる |

## 外へ出したものの行き先

- **時々しか要らない**ドメイン知識や複数ステップのワークフロー → **skill**(都度ロード)、
  または **`@path` で参照する別ファイル**にして必要時だけ読む。
  CLAUDE.md には1行ポインタだけ残す。
- **毎回・決定的に必須**(モデルの裁量に委ねられない検査) → 散文でなく **hook** に。
  指示は助言にすぎないが、hook は強制される。

## 点検手順

1. **棚卸し**: 対象ファイルと import 先を読む。現状の行数と概算トークン数を「before」として記録する。
2. **分類**: 各行を上の判定基準で仕分ける — 残す(不変則・広く適用・導けない)/ skill か `@` ファイルへ移す / hook へ転換 / 削除(導ける・陳腐・コード矛盾・重複)。
3. **書き直し**: 残すものを:
   - 命令形・具体・検証可能(「X を変えたら Y も変える」。「X に気をつける」は不可)。
   - 未定義語・勝手造語を排除 — [[no-neologism]] を併用。
   - 一文一行・段落間は空行。
   - トピックで節分けし、短い見出しを付ける。
   - 強調(IMPORTANT / YOU MUST)は、実際に無視される少数の規則にだけ使う。全部が叫べば何も効かない。
4. **報告**: 変更サマリ(各削除の理由 — 導ける/陳腐/重複/skill へ移動/hook へ転換)と before/after の行数・トークン数。

## ガード

派生可能・誤りと確証できないものは消さない。
迷ったら落とさず、ユーザーへフラグとして示す。
安全策が欠けるより、ファイルが少し長い方がまし。

## 推敲が要るサイン

- 規則があるのに agent が同じ間違いを繰り返す → ファイルが長すぎて規則が埋もれている。刈り込む。
- ファイルに**既に書いてある**ことを agent が訊いてくる → 表現が曖昧。
  行を足すのでなく、その行を書き直す。

## 併用

- `meta/empirical-prompt-tuning` は指示文の実測 QA ループ(operator 起動)。この skill は直接編集の手順 — スリム化後に [[empirical-prompt-tuning]] を回して、まだ機能することを確かめる。
- 未定義語の点検は [[no-neologism]] に委譲する。

Anthropic の "Write an effective CLAUDE.md" の指針に基づく
(<https://code.claude.com/docs/en/best-practices>)。
