# Model Routing (委譲ポリシー)

トークンを削る一番効くレバーは「安価モデルに委譲できる仕事を委譲する」こと。

## 一般方針

具体的なモデル名ではなく、能力ティアで考える。

- **強ティア(メインセッションに残す)**: 設計判断、論証、曖昧さの解消、
  トレードオフの検討、最終レビュー。ここで能力を下げると質が落ちる。
- **安ティア(サブエージェントへ委譲)**: 機械的・量的な作業、および読み取りの重い
  広域探索・調査。ここは品質が能力に縛られないので、強ティアに払うのは無駄。

形はいつも同じ。**重い思考は強いモデルに残し、機械的・量的な仕事を安価な
サブエージェントへ出す。** プラットフォームごとの機構(サブエージェントの起動方法、
セッション途中でモデルを切り替えられるか)は異なる。後述の「プラットフォーム実装」を見よ。

## 委譲の判断基準

委譲する:

- 作業を一文で明確に指示でき、出来上がりの正否が機械的に判定できる。
- ファイルをまたぐ探索だが、欲しいのは結論(該当箇所/要約)だけ。
- 同じ操作の反復で、量が多い。

メインセッションに残す:

- 何をすべきか自体を決める段階(設計・方針)。
- 失敗のコストが高く、判断に文脈の含みが要る。
- 短くて委譲のオーバーヘッドの方が高い。

両方の signal が立つとき(機械的だが量が少ない等)は、委譲のオーバーヘッド
(委譲文の作成・起動・結果回収)が実作業を上回るならメインに残す。「機械的だから」
だけを理由に些末なタスクを委譲しない。

## 文脈の最小化

委譲はトークンを二重に削る。安ティアが作業を行い、**かつ**読み取りの重い中間文脈が
メインセッションに入らない。後者の節約を得るには、委譲先が読んだものの全文転記ではなく
**結論だけ**を返す必要がある。

委譲時は、次の3点を必ず埋める。埋めないと委譲先が裁量で補い、結果がぶれる。

- **対象範囲**: 何を / どのファイル群を扱うか。除外条件も書く。
- **期待する成果物**: 何が出来上がっていればよいか。
- **返してほしい形**: 差分 / 出典付き要約(`path:line`) / 変更ファイル一覧 のどれか。

委譲先には結論だけを返すよう指示し、大きな Markdown は全文でなく節単位(mdidx)で
読ませる。委譲先自身の文脈も安く保つ。

## プラットフォーム実装

### Claude Code

Claude Code はメインセッションのモデルを skill/hook から自動切替できないので、委譲は
Task/Agent ツールで明示的に行う。本リポジトリは委譲先 agent を2つ同梱する(`agents/`
で定義、`~/.claude/agents/` に配置)。`subagent_type` に名前を指定して呼ぶ。

- **`bulk-edit`(安ティア)**: 指示が明確で判断の要らない作業 —
  一括リネーム、整形、定型の文字列置換、テンプレ展開、反復編集。
- **`search`(中ティア)**: コードベース探索・要約・調査(fastcontext 実行含む)、
  中規模で筋の見えている実装。

委譲文の例(bulk-edit へ一括リネーム):

> subagent_type: bulk-edit
> 対象範囲: リポジトリ全体の識別子 `foo_bar`(単語境界一致のみ、部分一致・コメント内は除外)
> 作業: `fooBar` へ機械的に置換。挙動は変えない。曖昧な箇所は置換せず保留して報告。
> 返す形: 変更ファイル一覧と置換件数の要約(差分は不要)。

委譲文の例(search へ調査):

> subagent_type: search
> 対象範囲: レート制限の実装箇所(ミドルウェア/デコレータ/設定/カウンタ)
> 期待する成果物: 方式・制限値・適用範囲・主要ファイルの要約
> 返す形: 出典付き(`path:line`)の簡潔な要約。コード全文転記は不要。

### Codex

agent 定義(`model:`)は Claude Code 固有。Codex では本 skill の「一般方針」と
「委譲の判断基準」に従い、Codex 自身のモデル設定(`/model` 等)で重い作業と軽い作業を
使い分ける。
