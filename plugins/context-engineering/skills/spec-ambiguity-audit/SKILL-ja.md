# 仕様書曖昧点監査 (Spec Ambiguity Audit)

文脈理解度の低いモデルが仕様書を素で読むと、著者が読み飛ばす箇所に気づく。
文中の言葉だけから全部を解決しなければならないからだ。この skill の価値はそれだけ
にある。**下位モデルの「文脈を知らないこと」を逆手に取る。** プロジェクトの要約を
与えない、想定される疑問に先回りして答えない、事前情報を与えない。プロジェクトを
説明した瞬間、この skill を有用にしている性質そのものが消える。

引き換えのコストは、挙げられた疑問の相当割合が「文書のすぐ近くに答えが書いてある」
ケースだということ。下位モデルは答えを引用しながら、それでもなお疑問として挙げて
くる。このフィルタ工程は省略できない。省略すると、有用な信号がユーザー自身に
再検証させるノイズに変わる。

## 二段構成

**第1段(安価モデル、素読み)**: 安価モデルに、文書のパスだけを渡してこう指示する —
これから実装するエンジニアとして読み、複数の実装のどれが意図されているか判断できない
箇所をすべて挙げよ、答えは推測するな。すべての項目に行番号の引用を必須にする —
これは省略できない。第2段のフィルタが行番号に依存するため。正確なプロンプトは
`references/prompt-template.md`(日本語版含む)を、この段をどう起動するかは下記
「プラットフォーム別の実装」を見よ。

**第2段(機械的フィルタ、モデル呼び出しなし)**: 引用された行範囲を集め、ソートして
重複・隣接する範囲をマージし、`scripts/extract_context.py` で該当範囲だけを元ファイル
から抽出する。そのうえで、メインモデルとして**抽出した範囲だけ**を読み(文書全体は
読まない)、各項目を分類する。

- **有効**: 抽出範囲を読んでも答えがない。残す。
- **既出解決**: 抽出範囲に答えがある。下位モデルが引用したまさにその文であることも
  ある。除外する。
- **ノイズ**: 誤読、または文書が定義する義務のない事柄についての疑問。除外する。

生き残った項目だけを、引用つきでユーザーに報告する。何をどんな理由で除外したかも
添えると、残ったリストへの信頼につながり、コストも小さい。

## フィルタをモデル呼び出しにしてはいけない理由

以前試した版では、第2段を**同じ下位モデル**に自分の引用文脈を読み直させて自己
フィルタさせた(「この行はもう疑問に答えていないか」)。結果、何も除外しなかった。
自分が直前に引用した文がまさにその答えであるケースでさえ、である。かわりに
より細かく限定した疑問を捏造して項目を生き延びさせた。自分の過去の出力を採点する
よう頼まれたモデルは、それを弁護する方向にアンカリングされ、修正する方向には
働かない。フィルタを、リストを作ったのと同じモデル(自分自身であれ新規セッションで
あれ)に通してはいけない。grep して読むという、結果に利害のない手段を使う。

## 抽出前にマージする理由

引用された行範囲は偏って集まりやすい — 複数の疑問が同じ段落や近接する2箇所を
指すことがよくある。マージせずに**各引用ごとに**固定マージンで独立に抽出すると、
元ファイルの行数を超える量を抽出してしまうことがある(実測: 673行のファイルから
708行を抽出)。読む前に必ず範囲をソートしてマージすること — `extract_context.py`
がこれを行う。

```bash
python3 scripts/extract_context.py <file> --margin 10 \
  --range 62 --range 316-318 --range 300-301
```

第1段で挙がった引用範囲は必ず一度の呼び出しでまとめて渡すこと。そうしないと
個々のコマンド内でしか重複排除が効かない。

## 機械的フィルタが割に合わないとき

短い文書(目安: 500行未満)で引用が文書の大半に散らばっている場合、マージ後の
抽出範囲がファイルの大半を覆ってしまうことがある。その場合はスクリプトを使わず
文書を一度通読した方が早い。この skill の効果は文書が長く、引用が疎らであるほど
大きい(数百〜数千行の仕様書・計画書が最も効く)。

## 引用箇所の近傍だけでは足りないとき

まれに、本当の答えが別の節にあることがある(例: 「リベース」節で挙がった疑問の
答えが100行以上離れた「自動保存」節にある)。検証では既出解決だった項目のうち
おおよそ6件に1件程度だったので、これは例外処理として扱う。抽出範囲を読んでも
解決せず、他所に答えがありそうだと感じたら、文書全体を読み直すのではなく、
その答えを狙って検索する。

## precision の数字を報告するときの注意

有効件数/総数の比率をユーザーに報告するなら、それが何を測っているかを明示する。
これは下位モデルの判断力と同じくらい、文書の成熟度を反映する。未記述の節が多い
粗い草稿は、ほとんど全部が本当に未定義なので precision が高く出る — これは監査の
精度が上がったのではなく、文書が薄いだけである。練られた仕様書で50%を超える
precision が出た方が、はるかに意味のある信号である。

## プラットフォーム別の実装

### Claude Code

第1段は `Agent` ツール、`subagent_type: search`(または素の `Agent` 呼び出し)、
`model: haiku`(またはもっとも安いモデル)で起動する。送るプロンプトは
`references/prompt-template.md` を見よ。

### Codex

Codex にはサブエージェントごとにモデルを切り替える機構がない。別の Codex セッションを
`/model` でもっとも安いモデルに切り替え、`references/prompt-template.md` の第1段
プロンプトを貼り付けて実行し、結果のリスト(引用つき)をメインセッションにコピーして
第2段に渡す。安いモデルが使えない場合は、コスト削減のねらいは諦めて、プロジェクトの
文脈を読み込んでいない新規セッションでメインモデルに素読みさせればよい —
この skill の価値は「安価なモデル」そのものではなく「素読み」から来る。
